# Beepo's Kitchen — Playtest Handoff (2026-07-05)

**Scope:** three features + one hardening batch, all decided after a full in-browser playtest (6 rounds, all three verdict tiers) and a code/art audit. Everything here is **pre-reskin work on the current SVG art** and slots in as **Batch 4.5** before the Batch 5 device gate (PLAN.md §8) — *except* §5, which changes the art-track asset manifest and **must be locked before any mass ComfyUI generation**.

**Build order:** B#14 (eating fix) → B#15 (equation strip) → B#16 (recipe book) → H#… sweep. B#15's strip component is reused by B#16's cards, and H#1–H#4 touch the same functions as B#14, so fold them into the same pass where convenient.

## 1. B#14 — Beepo eats the bowl: the vessel/food split

### 1.1 Root cause

- `bite()` scales `#dishBig` wholesale: `game.js:666-667` (`translate(-50%,-50%) scale(1-bitesDone*0.19)`), and `finishFeeding()` zeroes it (`game.js:675`).
- The white plate (`#plate`, a div — `index.html:646`) survives, but any bowl/glass/jar/plate *drawn inside the dish SVG* shrinks into Beepo with the food.
- Affected: all 103 generated living bowls (`funBowlSVG`, `game.js:494-511` — 129 catalog combos minus the 26 `DISH_LOOKS` entries), every `dish-bowl` look, the smoothie glass, jam jar, sundae glass, cake/custard/pancake/birthday plates, popsicle sticks — and the birthday dish's **lit candle**.

### 1.2 The layer contract (this is also the reskin contract — see §5)

Every dish render becomes **four layers**, composed by `reveal()` (`game.js:524-526`) inside the existing `#dishBig` svg:

```html
<svg viewBox="0 0 100 100">
  <g class="lyr-vessel">…</g>        <!-- bowl body/plate/glass/stick: NEVER eaten -->
  <g class="lyr-food">…</g>          <!-- everything edible incl. food toppers: depletes -->
  <g class="lyr-vessel-front">…</g>  <!-- thin front rim/lip drawn OVER food (depth fake) -->
  <g class="lyr-fx">…</g>            <!-- steam, sparkles, stink: never scaled by bites -->
</svg>
```

Three layers of vessel/food aren't enough as two: the front-overlay layer exists because three looks draw vessel geometry *in front of* the food to fake depth — the banana-split boat (`index.html:168`, painted last), the taco shell (`index.html:332`), and `funBowlSVG`'s front soup lip (`game.js:505`, the whole point of the B#13 half-dipped look). A two-sprite contract can't show food sitting *down inside* a vessel.

### 1.3 Symbol change: split `dish-bowl`

`dish-bowl` (`index.html:145-151`) currently bakes the food into the vessel: bowl body (146), stripe (147), foot (148), **soup-surface ellipse in `currentColor` (149)**, steam wisps (150).

- New symbol **`dish-bowl-shell`** = body + stripe + foot only.
- The soup ellipse moves out as its own symbol (**`dish-soup`**, `currentColor`) → food layer.
- Steam wisps → fx layer (an emptied bowl must stop steaming). Same for the steam baked into `dish-pie` (`index.html:205`).
- The front-lip path `funBowlSVG` already draws (`game.js:505`) becomes the `lyr-vessel-front` content for bowls.

Do **not** try the naive fix ("keep `#dish-bowl` static, scale the rest") — audit showed it breaks everything: the `apple`/`banana` looks are `dish-bowl` *only* (nothing left to scale → 5 bites with zero visible change), `choco`/`icecream`/`rainbow` would deplete only their topper/sparkles while the soup stays full, and the front lip would shrink away from the static rim mid-eat.

### 1.4 `funBowlSVG()` restructure (covers 111/129 dishes)

`funBowlSVG` (`game.js:494-511`) already composes in separable strings — restructure the generator to emit the four groups:

- `lyr-vessel`: `<use href="#dish-bowl-shell">` tinted as today.
- `lyr-food`: soup ellipse (`dish-soup`, blend-tinted) + the scaled half-dipped ingredient `<use>`s (`game.js:502`) + the soup smiley (`g-face`, `game.js:507`) + highlights.
- `lyr-vessel-front`: the front lip path (`game.js:505`) — **vessel-tinted, not food**: it reads as the bowl's inner wall. (Alternative: treat the lip as soup surface and put it in `lyr-food`, sinking it with the level. Try vessel-front first; pick whichever reads better at 300px.)
- `lyr-fx`: steam + verdict toppers `top-sparkles`/`top-stink` (`game.js:508-509`).

The 8 `DISH_LOOKS` entries built on `dish-bowl` (`apple+berry`, `apple+banana+berry`, and the singles `apple`/`banana`/`choco`/`icecream`/`worm`/`rainbow`, `game.js:43-57`) go through the same composition path. Topper classification for these:

| Topper | Layer |
|---|---|
| `top-cream` (`index.html:395-400`), `top-fruit` (383-387) | **food** — they deplete |
| `top-sparkles` (368-372), `top-stink` (402-405) | **fx** — never bitten |
| `top-worm` (360-366), `top-noodles` (378-381 — includes chopsticks, i.e. a utensil) | yucky-only; refuse path, never eaten — layer irrelevant in v1 |

### 1.5 Bespoke dishes: `eatWhole` tag + cheap regroups

Add an eat-style classification (simplest: a `Set` of combo keys, or a third element on `DISH_LOOKS` entries):

- **`eatWhole` (v1 default for all bespoke looks):** burger, pie, donut, cake?, icesand, cookiemush, cone, pancakes?, taco (yucky anyway), pops?, split, sundae, smoothie, jar, custard, bday. Keep today's shrink, but move the transform-origin toward Beepo (he stands at the plate's lower-left — origin ≈ `25% 65%`) so the food visibly shrinks *into his mouth* instead of into nothing.
- **Cheap mechanical regroups (optional v1.1 of this fix, ~30 min total):** seven looks have vessel and food as *already-separate* elements and just need `<g>` wrappers inside the symbol — cake (plate ellipse `index.html:214`, drawn LAST — z-order care), bday (plate 296; candle 300-302 becomes a **prop** that stays on the plate — Beepo stops swallowing a lit candle), pancakes (plate 244, already drawn first), custard (plate 289), pops (stick path 218 → the charming end state is three bare sticks), sundae (glass 175-177), split (boat 168 → vessel-front).
- **Do NOT redraw `dish-jar` (153-159) or `dish-smoothie` (280-286):** their vessel and liquid are a *single path* (jar body rect 154 IS the jam; cup body path 281 IS the liquid). A proper split needs a redraw — throwaway work, the reskin replaces them. They stay `eatWhole` until the reskin, where drink/jar vessels get the proper glass-outline + interior-fill + level-drop treatment (§5).

### 1.6 Eating mechanics changes (`bite()` / `finishFeeding()`)

Cadence is **untouched** (480/1050 ms, `game.js:670` — but see H#3's tier rebalance). All new juice is fire-and-forget; nothing blocks or queues.

- `bite()` retargets the per-bite transform from `#dishBig` to `dishBig.querySelector('.lyr-food')`, with `transform-origin: 50% 85%` (bottom-center) → the food **sinks into the bowl** as it shrinks. `#dishBig` itself keeps only the static centering transform.
- **Chunk-fly:** on bites 1, 3, 5 clone one ingredient icon out of the food layer as a `document.body` flyer (existing pattern, `game.js:322-338`) aimed at `revealBeepo`'s mouth rect, duration ≤ 400 ms so even fast-tier bites never overlap-queue. Remove that ingredient's `<use>` from the food layer when its flyer launches — depletion reads as *discrete, countable* events, which is what a 2-year-old can track (and it IS B#12's full/half/nearly-gone, expressed with zero new art).
- `finishFeeding()` (`game.js:672-688`): zero out **only** `.lyr-food`; add `.clean` to `.lyr-vessel` → CSS shine-sweep keyframe + `sDing()`. The persistent empty bowl is the punchline — "all gone!" is a meal script every toddler knows, and it makes the existing `'More please?'` line (`EAT_END`) land.
- `eatWhole` dishes: same code path, but the food layer is the whole dish and there's no `.clean` beat (crumbs remain the residue).
- Refuse path (`refuseDish`) is untouched — worm dishes are never eaten (all 37 are `yucky`), so they need none of this.

### 1.7 CSS additions (`styles.css`)

```css
#dishBig .lyr-food { transform-origin: 50% 85%; }
#dishBig .lyr-fx, #dishBig .lyr-vessel, #dishBig .lyr-vessel-front { /* never transformed by bites */ }
.lyr-vessel.clean { animation: vesselShine .7s ease-out; }
@keyframes vesselShine { /* white gradient sweep across the bowl, like the cloche shine */ }
```

(Implementation note: per-bite transforms on `.lyr-food` should keep using inline WAAPI/style like today's code — the classes are for the end-state and shine only.)

## 2. B#15 — Recipe equation strip (ingredients = dish)

**What:** upgrade the existing mini-icon row under the dish (`#dishIngs`, filled at `game.js:527`) into `[🍎 chip] + [🥚 chip] = [mini-dish chip]`.

**Why it works for the audience:** the 2yo reads it as a picture-story; for the 4-5yo the `+`/`=` structure is proto-math and the thing that turns mashing into intent ("I want the burger, so cookie + egg"). It's also the single best trainer for the recipe book.

### 2.1 Implementation

- **Shared component** (also used by B#16 cards):
  ```js
  function recipeStripHTML(ids, dishMarkup, size=34)
  ```
  Renders: ingredient chip per id, `ui-plus` between chips, `ui-equals`, then a mini dish chip (~44px) that **reuses the exact inner markup of `#dishBig`** (free — `findDish` already produced it).
- **Two new house-style symbols**, code-drawn per B#11 (no art pipeline, reskin-proof): `ui-plus`, `ui-equals` — chunky `#4A3B47` rounded strokes, small, decorative weight.
- **Key discipline:** build the strip from the **deduped, sorted catalog key ids** — *not* raw `potItems`. Today `game.js:527` renders duplicates ("apple apple banana" for a dish keyed `apple+banana`), which would teach a false rule that quantity matters. Refactor `findDish()` (`game.js:482-490`) to also return its `key`, and carry `key` + `ids` through `pendingResult` (`game.js:521`).
- **Timing — zero added loop time.** The strip rides the existing `#dishIngs` fade slot (visible from ~1.4s after `.revealed`, `styles.css:114-115`) *during* eating. It is **not** a beat between `finishReveal()` and `beginFeeding()` — inserting 2-3 s into a ~2.4 s eating loop would slow the core repetition by 50%+.
- **Pre-readers read time, not left-to-right:** stagger each chip with a pop-in keyframe (`scale 0 → 1.15 → 1`), ~150 ms apart, so the strip performs "apple… and egg… made THIS!" as a sequence. Delays land ~1.2-1.8 s after `.revealed`.
- **Silent** (or one soft tick total): first chomp hits ~2.2 s after `.revealed` and fast-tier chomps run at 480 ms — per-chip motif sounds would smear into the chomp track.
- **Chips are tappable:** `pointerdown` on an ingredient chip replays its boing + motif (mirrors shelf behavior; it coinciding with the plate's bonus-bite handler at `game.js:689` is fine — the two effects stack nicely).
- **Refuse path:** keep the strip, add a wiggle animation to the worm chip — teaches *which* ingredient caused the gag without reading as blame.

## 3. B#16 — Recipe book

### 3.1 The one hard design rule

**Scrapbook, not checklist.** 129 dishes, 84 near-identically-named triples: an exhaustive grid of "?" silhouettes is 87 locked boxes = 87 tiny "no"s — the game's first fail state. Therefore: **no global counter, no progress bar, no percentages, no completion reward, never render the full undiscovered set.**

### 3.2 Persistence

```js
// localStorage "bk-dishes"
{ "v": 1, "d": { "apple+banana": { "t": 1751712345678, "c": 3 } } }
// t = first-discovered epoch ms (parent-facing "first cooked"), c = times cooked
```

- Keys are the **sorted combo keys** `findDish()` already computes (`game.js:483-484`) — stable across the reskin (ids never change; display names might).
- `recordDiscovery(key)` returns `{isNew}` and is called from **`finishReveal()`** (`game.js:572`) — discovery = the moment the dish is *seen*, not cooked. **Yucky dishes count**: the refuse gag is the best outcome in the game and worm cards are the ones a 4yo shows a parent.
- Wrap all access in try/catch like `bk-music`/`bk-muted` (`game.js:130,198`).
- Store **no art and no names** — cards regenerate from the key via `findDish`. Refactor `findDish(ids)` to take an ids array instead of reading the `potItems` global, so the book can call it per card.

### 3.3 The book UI

- **`#bookBtn`:** kid-sized (~64 px), **top-left** — deliberately opposite the 40 px parent utility buttons (mute/music, top-right, `index.html:505-508` region). Hidden until the first discovery exists.
- **Opens only from the kitchen screen** (ignore taps while `#overlay` is showing) — the cook loop stays unbreakable.
- **Content:** reverse-chronological scroll of **discovered dishes only** (newest first — toddlers want the thing they *just* made), one big card per dish: large dish render (the four-layer markup from B#14) + `recipeStripHTML` + small dish name and "first cooked" date as the parent layer.
- **Teasers:** at most **3** "quest teaser" cards appended at the end — real ingredient chips, sparkling `?` dish chip (`top-question` at `index.html:374-376` is currently dead — wire it here), each chosen to **share ≥1 ingredient with the last-cooked dish**, preferring 2-ingredient combos and bespoke looks. An actionable invitation a 5yo can execute at the shelf, not pressure.
- **Card tap:** boing + the dish's verdict sound + Beepo peeks in with the matching `DISH_LINES` one-liner (`game.js:294-300`). Pure replay-of-joy; no navigation depth.
- **Never-stall:** giant X button AND tap-outside AND a ~30 s idle auto-close (extend `armIdle`/`currentTarget()`, `game.js:232-245` — while the book is open the guide hand points at the X, then the timer closes it).
- **Performance:** lazy-render (discovered + 3 teasers only). 129 inline SVGs at once will hurt the cheap-tablet target (PLAN.md §10).

### 3.4 The "NEW!" moment

Wordless discovery stamp — the stamp IS the book onboarding (no tutorial possible at this age):

1. New symbol `ui-stamp-new`: gold starburst rosette + sparkles, tiny "NEW!" text inside as parent garnish only.
2. **Yum/magical path:** ~900 ms after the tada in `finishReveal()`, stamp slams onto the plate corner with a rubber-stamp squash (`scale 1.6 → 0.9 → 1`) + a **distinct** sound (low thunk + chime — NOT `sTada`, which already fired; discovery must be audibly different from the every-round celebration).
3. During `finishFeeding()`'s end beat, the stamp shrinks and **flies to `#bookBtn`** (existing flyer pattern, ≤600 ms); the button boings and gains a badge dot. That flight wordlessly teaches where the book lives.
4. **Refuse path:** delay the slam to the giggly-recovery beat (~3.5 s, `game.js:631`) so it never steps on the faint gag.
5. **Timing budget:** auto-restart fires 5.2 s / 8.2 s (`game.js:687/633`) — the whole sequence must complete by ~4.5 s. It does (slam +0.9 s, flight ≤0.6 s), but H#2 (idle-based restart) adds slack anyway.

### 3.5 Defer (v2 / post-reskin)

Ingredient-filter tabs; page-flip metaphor; **cook-from-book** (tap a card to preload the pot — genuinely great 5yo intent play, but it bypasses the shelf and touches the core loop; needs its own playtest); any second-level card detail view; any completion rewards (never, per §3.1).

## 4. H# — Playtest hardening (Batch 4.5)

Ordered by real-toddler impact. H#1-H#4 are P0 and should land with B#14; H#2 and H#3 change locked-at-gate timings so they **must precede the Batch 5 sign-off**.

**H#1 — Post-reset input shield (the worm tap-through).** *Playtest-confirmed twice in six rounds.* The circular `#againBtn` overlaps the row-3-center shelf cell — which is the **worm** (ingredient order `game.js:13-23`; geometry: on a 390×844 phone the button's ~96 px circle sits ~90% on the worm cell). Because the overlay closes instantly (auto-restart `display:none`) and the shelf listens on `pointerdown` while the button fires on `click`, a double-tap or a tap racing the auto-restart lands on the worm: the next pot silently becomes yucky — a hidden consequence the child never chose, i.e. the game's one real fail state. **Fix (~5 lines):** record `overlayClosedAt` in the `againBtn` handler (`game.js:710`); `addIngredient()` (`game.js:315`) and the pot handler (`game.js:410`) ignore input for ~400 ms after it. No shelf dimming (that was B#0's lesson) — just swallow the first ghost tap. Optionally add `pointerdown` parity on `#againBtn` for snappier feel.

**H#2 — Auto-restart must be idle-based.** `finishFeeding` schedules `againBtn.click()` at a flat 5200 ms (`game.js:687`), `refuseDish` at 8200 ms (`game.js:633`), regardless of input — the screen can vanish mid-tap (out-raced the playtest twice), and the in-flight tap then lands on the shelf (H#1's race, but timer-triggered). **Fix:** re-arm the restart timer on any overlay `pointerdown` (same pattern as `armIdle`). Keeps never-stall; stops yanking the screen from an engaged child.

**H#3 — Restore the reward gradient; taps must never shorten the show.** `'vor'` fires for ANY 2+ ingredient dish (`game.js:519-520`), so the burp-confetti maximum celebration is the *default* and the dish blurs away in ~2.4 s — too fast for a toddler to track, and magical dishes barely out-celebrate banana+egg. Worse, plate taps *increment* `bitesDone` (`game.js:662,689`): a child who joins in ends the party sooner. **Fix:** (a) `vor` = 3-ingredient or magical only (one line); (b) split `bite()` into `advanceBite()` (auto-rhythm, increments `bitesDone`) and `bonusChomp()` (tap: chomp sound + Beepo squash + crumbs + food wobble, does NOT increment) — tapping adds spectacle, never subtracts bites. *(Optional, +~30 lines, pairs with B#16 teasers: "Beepo's craving" — a per-session random ingredient shown wordlessly in his idle thought bubble; dishes containing it upgrade one tier with bonus confetti.)*

**H#4 — Whole overlay = tap target.** Only `plateWrap` counts bites (`game.js:689`) and only the cloche lifts the cover (`game.js:570`); ~70% of the fixed full-screen overlay is dead. There's exactly one meaningful action at any moment, so: covered state → any overlay tap lifts; feeding → any overlay tap = `bonusChomp()`; the 3.4 s revealed-but-not-yet-feeding window → a tap fast-forwards to `beginFeeding()`. Conflict-check done: `#againBtn` is `pointer-events:none` until shown; `revealBeepo` sits inside `plateWrap`.

**H#5 — Shelf feedback during cook; no vanishing flyers.** `addIngredient()` early-returns on `cooking` *before* the boing/motif (`game.js:316-319`) — every tap during the 2 s cook is silently dead (the exact violation B#0 was written to kill). And a flyer mid-air when cooking starts evaporates at landing (`game.js:334`) — the child watched their apple fly in and it's not in the meal. **Fix:** during `cooking`, still boing + motif on the shelf (just don't spawn a flyer); flyers that land mid-cook `bounceOff()` the rim instead of vanishing.

**H#6 — Stage Beepo is tappable.** The most attractive object on the kitchen screen has no handler (`#beepo`, `game.js:177`). Five lines: `pointerdown` → hop (pattern `game.js:383`) + random `say()` + `setExp('yum'|'wow')` + giggle tone.

**H#7 — Master compressor.** Every `tone()` connects straight to `actx.destination` (`game.js:76`); overlapping flyers are a design goal, so a two-handed mash stacks 10+ oscillators + music into hard clipping — harsh noise, the opposite of the sensory brief. **Fix:** one shared `DynamicsCompressorNode` → destination at startup; route `musicGain` and all `tone()` gains through it.

**H#8 — `prefers-reduced-motion` is a blanket nuke.** `styles.css:286-288` collapses ALL animations to .01 s/1 iteration: (1) the wordless cue system (potGlow, hintPulse, againBtn pulse, guide-hand bounce, flame flick) freezes — exactly the users who need calm motion lose the game's only instruction channel; (2) the cloche transition collapses but `liftCover` still waits a hardcoded 2100 ms (`game.js:568`) — a dead stall; (3) the WAAPI animations (`element.animate()` flyers etc.) aren't governed by CSS rules anyway. **Fix:** replace the blanket rule with a curated list (kill shakes, large translations, confetti spin; keep gentle cue pulses), and when `matchMedia('(prefers-reduced-motion)')` matches, shorten the `finishReveal` wait to match the collapsed transition.

**H#9 — Landscape/short viewports clip the again-button off-screen.** Overlay column ≈ 493 px in a non-scrolling fixed overlay; on a ~390 px-high landscape phone the button is below the fold and the only recovery is the auto-restart. Portrait lock only arrives with Capacitor (Batch 6), and the web build is what's being playtested. **Fix:** `@media (max-height: 560px)` → shrink `#plateWrap` (300→~210 px) and margins so the button always fits.

**H#10 — Minor sweep.** (a) `#musicBtn` lies while muted (toggles "on", `startMusic` silently returns — `game.js:157`): auto-unmute on enable, or render it disabled while muted. (b) Guard the `visibilitychange` → `startMusic` path (`game.js:170`) behind a "title tapped" flag so no AudioContext is constructed pre-gesture. (c) Dead attract branch: `say('Cook it!')` (`game.js:254`) is unreachable (auto-cook starts cooking within 4.2 s < the 8 s idle, and the `cooking`/overlay guards at `game.js:251` then suppress the attract line) — remove, or better: make `currentTarget()` point the guide hand at the pot during the pre-cook window (`game.js:242`) to teach the tap-to-cook-sooner accelerator. (d) `.floatItem` still has `cursor:pointer` + `:active` scale (`styles.css:55-56`) from removed tap-to-remove — delete the affordance.

## 5. Art-track manifest changes (LOCK BEFORE MASS GENERATION)

The B#14 layer contract changes what a "dish asset" *is*. If the ComfyUI batch renders 129 flat dish images, the eats-the-bowl bug is baked in permanently. Updates to PLAN.md §3.2/§3.4/§3.7:

1. **`dishes.json` gains a `vessel` field** per dish: `bowl | plate | glass | cup | stick | none`. (Default for all 103 generated combos: `bowl`.)
2. **Vessels are a shared library, rendered once:** ~5-6 vessel sprites (plus per-vessel front-rim strip where needed — bowl lip, taco shell, split boat). Same square canvas and registration as dish renders. Document the **soup-surface registration line** per vessel (the SVG-era bowl uses y=48; see the comment at `index.html:351`) so food sprites sit correctly.
3. **Per-dish renders become food-only:** prompt phrasing gains *"food only, no bowl, no plate, isolated"* (audit the `image_prompt` fields — this folds into B#13's existing prompt-audit TODO). Food is alpha-matted to sit inside the vessel.
4. **Eating states (B#12) apply to the FOOD layer only, and only for the 92 edible dishes** — all 37 worm combos are `yucky` and never reach `bite()`.
5. **Manifest math:** base = 129 food renders + ~6 vessels ≈ **135** images (vs 258+ if every dish shipped vessel+food pairs). Eating states: **try the runtime treatment first** — with the vessel staying crisp and only the food sprite sinking/shrinking (B#14's mechanic), generic depletion may already read as cute; decide at the pilot. If not, add per-state food renders (`dish-X-food-1/2/3`) for hero dishes only (B#12 option c): worst case 92×2 extra ≈ 184 → still under the old ×3-everything plan (387) *and* only the food layer needs curation.
6. **Dead symbols — decide before the manifest locks:** defined but unreachable today: `dish-omelette`, `dish-spaghetti`, `dish-juice`, `dish-pizza`, `dish-muffin`, `top-sock`, `top-question` (B#16 wants this one for teaser cards), `top-drips`, `ing-carrot`, `ing-cheese`, `ing-sock`. Cheap wins to widen bespoke coverage: `egg+rainbow` = "Rainbow Cloud Omelette" → `dish-omelette` + `top-sparkles`; worm-noodle yucky looks → `dish-spaghetti`. Otherwise drop them from the reskin batch (~11 sprites of zero screen time).
7. **§3.5 exporter:** export the *split* symbols (`dish-bowl-shell`, `dish-soup`, per-layer groups) as separate linerefs so ControlNet gets vessel and food references independently.

## 7. Optional / flagged for Gustav

**O#1 — Robot TTS as the deferred-VO stopgap.** PLAN.md §9 calls missing voice the biggest pre-reader gap; recording is deferred. The Web Speech API (`speechSynthesis`) sidesteps recording, and a slightly synthetic voice is *in character* for a robot chef (pitch ~1.6, slower rate). Wire under the existing `playVoice()` choke point (`game.js:115-124`), precedence `VOICE_CLIPS > TTS > tone motif`, so recorded clips still drop in later. Highest-value lines: ingredient names on shelf tap, "All gone!", dish one-liners; `utterance.lang` supports sv-SE + en. **Ship-gates:** (a) voice quality varies wildly by device — capability-check and fall back to tones; (b) some Android TTS voices are network-backed, which would break the zero-network Kids-Category claim (§6.3) — iOS default voices are on-device, but verify inside the Capacitor wrapper and disable TTS if any network use is observed.

**Not done in this pass:** none of the above is implemented; this document + the PLAN.md deltas are the handoff. The working tree at `5acf0c4` is untouched apart from documentation.
